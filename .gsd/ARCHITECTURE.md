## Data Flow

1. **User Action:** The user adds a task in `DailyLogTab` or triggers a brain dump in `BrainDumpStation`.
2. **State Update:** The child component executes a callback passed down from `src/App.tsx`.
3. **App Orchestration:** `src/App.tsx` updates its React state (`items`, `collections`, etc.).
4. **Side Effects:**
   - React state changes trigger a `useEffect` that serializes the new state to `localStorage`.
   - AI requests post messages to `src/ai.worker.ts` in the background. The worker runs inference and posts the results back, triggering state updates in `App.tsx`.
5. **Re-rendering:** The updated state triggers a full re-render of `App.tsx` and all dependent subcomponents.

## Technical Debt

- [ ] **Monolithic `App.tsx` (2,655 lines):** The file houses global states, UI tab selections, procedural Web Audio generation, canvas drawing parameters, export handlers, and form states. This consumes massive context when modifying code and makes the file error-prone.
- [ ] **Prop Drilling:** Main feature components depend directly on callbacks passed down from `App.tsx` (e.g. `items`, `setItems`, `settings`, `setSettings`). Changes to state structure require modifying files across multiple layers.
- [ ] **Procedural Audio Synthesis Mixed with UI:** The `startAmbientAudio` logic generates white/brown noise buffers and fireplace crackling procedural scripts inside `App.tsx`. This makes the code hard to read and test.
- [ ] **Lack of Global State Manager/Context:** No React Context is utilized. Moving state to `src/context/` would decouple layouts from data and clean up `App.tsx` prop-drilling.
- [ ] **Inline Exporters:** HTML-to-Canvas and PDF conversion codes are inline in `App.tsx`.

## Conventions

- React components use PascalCase (e.g., `DailyLogTab.tsx`).

**Structure:**
- Feature directories group their own components (e.g., `src/features/focus/components/`).
- Global reusable items are grouped in `src/components/common/`.
- Business logic helpers are in `src/utils/`.
